## Patterns napi-rs

### 1. Class wrapper (stateful)

```rust
#[napi]
pub struct HoraCore {
    inner: hora_graph_core::HoraCore,  // ownership
}

#[napi]
impl HoraCore {
    #[napi(factory)]
    pub fn new_memory(config: Option<HoraConfigJs>) -> napi::Result<Self> {
        let config = config.map(Into::into).unwrap_or_default();
        let inner = hora_graph_core::HoraCore::new(config)
            .map_err(|e| napi::Error::from_reason(format!("{}", e)))?;
        Ok(Self { inner })
    }

    #[napi(factory)]
    pub fn open(path: String, config: Option<HoraConfigJs>) -> napi::Result<Self> {
        // ...
    }
}
```

### 2. Plain objects pour les DTOs

```rust
#[napi(object)]
pub struct HoraConfigJs {
    pub embedding_dims: Option<u32>,
    pub storage_type: Option<String>,
}

#[napi(object)]
pub struct EntityJs {
    pub id: i64,          // BigInt pas supporte partout → i64
    pub entity_type: String,
    pub name: String,
    pub properties: Option<serde_json::Value>,  // ou HashMap
}

#[napi(object)]
pub struct SearchHitJs {
    pub entity_id: i64,
    pub score: f64,
    pub name: String,
}
```

### 3. Float32Array pour les embeddings (zero-copy)

```rust
#[napi]
impl HoraCore {
    #[napi]
    pub fn add_entity(
        &mut self,
        entity_type: String,
        name: String,
        properties: Option<serde_json::Value>,
        embedding: Option<Float32Array>,  // zero-copy depuis JS
    ) -> napi::Result<i64> {
        let emb_ref = embedding.as_ref().map(|e| e.as_ref());
        let id = self.inner.add_entity(
            &entity_type, &name,
            properties.map(/* convert */),
            emb_ref,
        ).map_err(napi_err)?;
        Ok(id.0 as i64)
    }

    #[napi]
    pub fn vector_search(
        &self,
        embedding: Float32Array,
        k: u32,
    ) -> napi::Result<Vec<SearchHitJs>> {
        // Float32Array → &[f32] zero-copy
        let results = self.inner.vector_search(embedding.as_ref(), k as usize)
            .map_err(napi_err)?;
        Ok(results.into_iter().map(SearchHitJs::from).collect())
    }
}
```

### 4. Async pour les operations longues

```rust
#[napi]
impl HoraCore {
    #[napi]
    pub async fn dream_cycle(&self) -> napi::Result<()> {
        // napi-rs gere le spawn_blocking automatiquement
        // Mais attention : &self n'est pas Send
        // Solution : cloner l'Arc interne
        let inner = self.inner.clone_arc();
        tokio::task::spawn_blocking(move || {
            inner.dream_cycle()
        }).await.unwrap().map_err(napi_err)
    }
}
```

### 5. Error handling

```rust
fn napi_err(e: hora_graph_core::HoraError) -> napi::Error {
    napi::Error::from_reason(format!("{}", e))
}
```

## Pieges courants

### 1. BigInt vs Number
- EntityId(u64) > Number.MAX_SAFE_INTEGER = bug silencieux
- Solution : utiliser i64 (signe) et caster en JS number
- En pratique, les IDs < 2^53 sont suffisants
- Alternative : BigInt (mais moins ergonomique en JS)

### 2. &self vs &mut self
- napi-rs ne supporte pas &mut self dans les methodes
- Solution : RwLock interne (pattern deja choisi pour hora)

### 3. Memory leaks
- Les objets JS ne sont GC que quand plus references
- Les classes napi ont un destructor (`Drop`)
- Fournir un `close()` explicite pour liberer les ressources

### 4. Thread safety
- napi-rs execute les methodes sur le main thread par defaut
- Les methodes async sont executees sur un thread pool
- HoraCore avec Arc<RwLock<...>> est Send + Sync → safe
