# Phase v0.5 — Robustness (Storage page-based + WAL)

> Passer du format binaire simple (v0.1) au format page-based production-ready.
> WAL, mmap, crash recovery, compaction.

---

## Sous-phases

### v0.5a — Page Allocator + Freelist (2-3 jours)

- [ ] `storage/embedded/page.rs` : PageAllocator
- [ ] Allocation de pages (4KB defaut, configurable)
- [ ] Freelist : liste chainee de pages libres
- [ ] `alloc_page(type)` : prendre depuis freelist ou fin de fichier
- [ ] `free_page(page_num)` : ajouter a la freelist
- [ ] File header : page_count, freelist_page, freelist_count
- [ ] CRC32 sur chaque page header

**Tests :**
- Alloc 10 pages → page_count = 11 (header + 10)
- Free 5 pages → freelist_count = 5
- Re-alloc → prend depuis la freelist d'abord
- CRC32 mismatch → CorruptedFile error

### v0.5b — B+ Tree pour Entities (3-4 jours)

- [ ] `storage/embedded/btree.rs` : B+ tree on-disk
- [ ] Interior pages : separateurs EntityId + pointeurs child
- [ ] Leaf pages : entites triees par ID, chainables (next_leaf)
- [ ] `insert(entity)` : trouver la bonne leaf, inserer, split si plein
- [ ] `get(id)` : traverser l'arbre, 3 lectures max pour <10M entities
- [ ] `delete(id)` : marquer comme supprime, merge si page <50% remplie
- [ ] `scan()` : parcourir les feuilles sequentiellement

**Tests :**
- Insert 1000 entities → toutes recuperables par ID
- Insert ordonne et desordonne : meme resultat
- Split de page leaf : 2 pages, chainee correctement
- Delete + re-insert : pas de corruption
- Scan : retourne toutes les entites dans l'ordre des IDs

**Benchmark :** get by ID < 500ns (3 page reads via mmap)

### v0.5c — WAL (Write-Ahead Log) (3-4 jours)

- [ ] `storage/embedded/wal.rs` : WAL implementation
- [ ] WAL header : magic "WLOG", version, page_size, salt
- [ ] WAL frame : page_number, db_size, salt, checksum, page_data
- [ ] Write path : serialize → WAL frame → fsync
- [ ] Read path : check WAL index → fallback to main file
- [ ] WAL index (in-memory) : HashMap<page_num, frame_offset>
- [ ] Checkpoint : replay WAL frames → main file → truncate WAL
- [ ] Auto-checkpoint quand WAL > 1000 frames

**Tests :**
- Write via WAL → readable before checkpoint
- Checkpoint → data in main file, WAL truncated
- Corrupted WAL frame (bad checksum) → frame ignored
- WAL salt mismatch → WAL ignored (stale)
- Concurrent reads during WAL write → readers see consistent snapshot

### v0.5d — mmap Reader (1-2 jours)

- [ ] `storage/embedded/mmap.rs` : MmapReader
- [ ] mmap le fichier principal en lecture seule
- [ ] `read_page(page_num)` → &[u8] zero-copy
- [ ] Fallback `read()` pour WASM et petits fichiers
- [ ] Re-mmap apres checkpoint (fichier a grandi)
- [ ] Feature flag : `#[cfg(not(target_arch = "wasm32"))]`

**Tests :**
- mmap read == read() result (byte-identical)
- Re-mmap apres checkpoint : nouvelles pages visibles
- WASM fallback : read() fonctionne sans mmap

### v0.5e — Crash Recovery (2-3 jours)

- [ ] Sequence de recovery a l'ouverture (voir 03-storage-engine.md)
- [ ] Verifier magic + version
- [ ] Si WAL existe : scanner, verifier checksums, construire WAL index
- [ ] Pages partiellement ecrites : detectees par CRC32
- [ ] Tests avec injection de crashes

**Tests :**
- Crash pendant WAL write → frame partiel ignore
- Crash pendant checkpoint → WAL rejoue correctement
- Fichier sans WAL → ouvre normalement
- Magic invalide → InvalidFile error
- Double open → write lock protege

**Benchmark :** WAL recovery 1000 frames < 100ms

### v0.5f — Compaction (1-2 jours)

- [ ] `compact()` : incremental vacuum
- [ ] Deplacer les pages de fin vers les trous (freelist)
- [ ] Mettre a jour les pointeurs B+ tree
- [ ] Truncate le fichier
- [ ] Full vacuum optionnel : rewrite complet dans .hora.tmp + rename

**Tests :**
- Delete 50% des entities → compact → fichier plus petit
- Compact ne corrompt pas les donnees (roundtrip test)
- Full vacuum : fichier identique fonctionnellement

### v0.5g — Transactions multi-statement (2-3 jours)

- [ ] `begin_transaction()` / `commit()` / `rollback()`
- [ ] WAL savepoints : marquer le debut de transaction
- [ ] Rollback : discard WAL frames depuis le savepoint
- [ ] Commit : les frames deviennent permanents
- [ ] Isolation : les autres readers ne voient pas les frames non-commites

**Tests :**
- Begin → insert → rollback → entity non visible
- Begin → insert → commit → entity visible
- Transaction imbriquee → erreur (pas supporte en v0.5)
- Crash pendant transaction → rollback automatique

## Notes d'implementation

### Ordre de migration v0.1 → v0.5

Le format v0.1 (binaire simple, tout en memoire) est incompatible avec le format v0.5 (page-based).

Strategy : **export/import**
1. Ouvrir le fichier v0.1
2. Lire toutes les entites/edges/episodes en memoire
3. Creer un nouveau fichier v0.5
4. Inserer tout dans le nouveau format
5. Renommer

C'est acceptable car en v0.x, le format n'est pas encore stable.
