## Sous-phases

### v0.6a — SQLite Backend (3-4 jours)

- [ ] `storage/sqlite.rs` : impl StorageOps pour SQLite via rusqlite
- [ ] Feature flag `sqlite` : `dep:rusqlite`
- [ ] Schema SQL :

```sql
CREATE TABLE entities (
    id INTEGER PRIMARY KEY,
    entity_type TEXT NOT NULL,
    name TEXT NOT NULL,
    properties BLOB,          -- binaire compact (meme format que embedded)
    embedding BLOB,           -- f32 array brut
    created_at INTEGER NOT NULL,
    activation_data BLOB      -- ActivationState serialise
);

CREATE TABLE edges (
    id INTEGER PRIMARY KEY,
    source_id INTEGER NOT NULL REFERENCES entities(id),
    target_id INTEGER NOT NULL REFERENCES entities(id),
    relation_type TEXT NOT NULL,
    description TEXT,
    confidence REAL DEFAULT 1.0,
    valid_at INTEGER NOT NULL,
    invalid_at INTEGER DEFAULT 0,
    created_at INTEGER NOT NULL
);

CREATE TABLE episodes (
    id INTEGER PRIMARY KEY,
    source TEXT NOT NULL,
    session_id TEXT,
    entity_ids BLOB,          -- packed u64 array
    fact_ids BLOB,            -- packed u64 array
    created_at INTEGER NOT NULL,
    consolidation_count INTEGER DEFAULT 0
);

CREATE TABLE string_pool (
    offset INTEGER PRIMARY KEY,
    value TEXT NOT NULL
);

-- Index
CREATE INDEX idx_edges_source ON edges(source_id);
CREATE INDEX idx_edges_target ON edges(target_id);
CREATE INDEX idx_edges_valid_at ON edges(valid_at);
CREATE INDEX idx_entities_name ON entities(name);
CREATE INDEX idx_entities_type ON entities(entity_type);
```

- [ ] BM25 via **FTS5** (SQLite built-in) :

```sql
CREATE VIRTUAL TABLE entities_fts USING fts5(name, content=entities, content_rowid=id);
```

- [ ] Vector search : brute-force en Rust (lire les embeddings, cosine SIMD)
- [ ] Transactions : utiliser les transactions SQLite natives
- [ ] Mapping complet de toutes les operations StorageOps

**Tests :**
- Tous les tests StorageOps passent avec SqliteStorage
- FTS5 : "authentication" trouve les bonnes entites
- Transaction rollback : entite non visible apres rollback
- Concurrent readers (SQLite WAL mode)

## Notes

### Pourquoi des backends alternatifs ?

| Use case | Backend recommande |
|----------|-------------------|
| Application standalone, CLI, edge | **Embedded** (.hora) |
| Application existante avec SQLite | **SQLite** |
| Application multi-client, cloud | **PostgreSQL** |
| Tests unitaires | **Memory** |
| Browser (WASM) | **Memory** ou IndexedDB (futur) |

### Ce que les backends alternatifs ne supportent PAS

- Le format page-based .hora (specifique a embedded)
- Le WAL custom (SQLite et Postgres ont leur propre WAL)
- mmap (gere par le backend lui-meme)
- Le dream cycle et les bio-features fonctionnent sur tous les backends (calculs en memoire)
