# Phase v0.3 — Memory (ACT-R + Reconsolidation)

> Activation decay, spreading, fan effect, dark nodes, reconsolidation.
> Le cerveau computationnel de hora.

---

## Sous-phases

### v0.3a — ACT-R Base-Level Learning (2-3 jours)

- [ ] `memory/activation.rs` : struct ActivationState
- [ ] Approximation hybride Petrov 2006 (10 recents exacts + historique agrege)
- [ ] `compute_activation(now)` : calcul BLL avec stabilite numerique (log-sum-exp)
- [ ] `record_access(now)` : enregistrer un acces, evacuer vers historique si >10
- [ ] Dirty flag + cache du dernier score calcule
- [ ] Integration dans `HoraCore` : `get_activation(entity_id)` et `record_access(entity_id)`
- [ ] Side-effect : `search()` et `get_entity()` appellent record_access automatiquement

**Tests :**
- BLL apres 1 acces : activation > 0
- BLL decay : activation(t+1000) < activation(t)
- BLL multiple acces : plus d'acces = activation plus haute
- Petrov approximation : |exact - approx| < 5% sur 1000 acces
- record_access overflow : 11eme acces evacue le plus ancien vers historique
- Stabilite numerique : pas de NaN/Inf pour des deltas > 10^8 secondes

**Benchmark :** compute_activation() < 100ns, record_access() < 50ns

### v0.3b — Spreading Activation + Fan Effect (2-3 jours)

- [ ] `memory/activation.rs` : spread_activation()
- [ ] BFS sur les edges (outgoing + incoming optionnel)
- [ ] Fan effect : S_ji = S_max - ln(fan), inhibition si fan > e^1.6 ≈ 5
- [ ] Cutoff : arreter la propagation si activation < seuil
- [ ] Max depth configurable (defaut: 3)
- [ ] Spreading cache : stocker le dernier resultat pour eviter les recalculs

**Tests :**
- A→B : spread depuis A → B recoit de l'activation
- A→B, A→C, A→D : fan=3, activation propagee est positive
- A→B1..B10 : fan=10 > 5, activation propagee est **negative** (inhibition)
- Cycle A→B→A : ne boucle pas infiniment
- Depth limit : depth=2, A→B→C→D, D ne recoit rien

**Benchmark :** spread_activation(depth=3) < 1ms sur 100K nodes

### v0.3c — Reconsolidation Window (1-2 jours)

- [ ] `memory/reconsolidation.rs` : enum MemoryState {Stable, Labile, Restabilizing, Dark}
- [ ] Transition Stable → Labile sur reactivation forte
- [ ] Transition Labile → Restabilizing apres expiration fenetre (5h defaut)
- [ ] Transition Restabilizing → Stable avec boost FSRS
- [ ] Reactivation faible : juste record_access, pas de destabilisation
- [ ] Seuils configurables

**Tests :**
- Acces fort sur Stable → passe en Labile
- Labile pendant 5h → passe en Restabilizing
- Restabilizing pendant 6h → retourne Stable avec boost
- Acces faible sur Stable → reste Stable
- State machine : pas de transition invalide possible

### v0.3d — Dark Nodes (1-2 jours)

- [ ] `memory/dark_nodes.rs` : detection, silencing, recovery
- [ ] `dark_node_pass()` : scanner toutes les entites, marquer les sub-threshold
- [ ] Edge attenuation : edges d'un dark node multiplies par 0.1
- [ ] Recovery : reactivation externe forte → Labile (re-encodage)
- [ ] `search()` : exclure dark nodes par defaut, flag `include_dark`
- [ ] `dark_nodes()` : lister tous les dark nodes

**Tests :**
- Entite jamais accedee pendant 7j + activation < -2.0 → Dark
- Dark node invisible dans search() par defaut
- Dark node visible si include_dark=true
- Recovery : stimulus externe fort → sort du Dark
- GC eligibility : dark > 90j → candidate

**Benchmark :** dark_node_pass() < 50ms sur 100K entities

### v0.3e — FSRS Scheduling (1-2 jours)

- [ ] `memory/fsrs.rs` : retrievability formula, stability update
- [ ] `current_retrievability(now)` : R = (1 + F·t/S)^(-w20)
- [ ] `next_review_interval(r_desired)` : quand planifier la prochaine revue
- [ ] Integration dans total_activation : BLL + spreading + FSRS modulator
- [ ] Parametres w20 et desired_retention configurables

**Tests :**
- R(t=0) = 1.0 (juste apres la revue)
- R(t→∞) → 0.0
- Stability augmente apres chaque revue reussie
- next_review_interval croissant avec la stabilite
